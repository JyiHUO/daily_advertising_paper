早上好，今天是四月二十八号，星期二。先给你一个具体的数字，把今天的主判断先钉住：快手把双塔召回升级了一下，线上广告收入最高涨了百分之八点三六。这个数字在广告圈已经很夸张了，而且不是靠堆特征、堆算力堆出来的，是靠重新设计双塔和下游精排之间的关系拿到的。

所以今天这期，我想先说一句判断：今天这批论文最值得记住的一件事，是工业界正在认真地重写"召回和排序之间的那堵墙"。过去我们默认召回就该便宜、就该粗糙，排序才做精细活儿；现在越来越多的论文在说，这堵墙其实可以拆一部分，而且拆完之后系统整体会更协同。

这不只是快手在做。今天精读的两篇、加上候选里的多模态 Semantic ID、RoPE 改造、MTServe 这些工作，背后都是同一种思路：召回要更像排序，训练要更贴近在线，标签要更贴近真实行为。换句话说，广告系统的每一层正在互相靠拢。

先说今天要你记住的三件事。

第一件，是《Follow the TRACE》。它把点击之后用户的行为轨迹直接建模进 CVR 预估。第二件，是快手的 CS3，双塔召回的新方案，线上收入最高加了八个多点。第三件，是快手另一篇 Adaptive Semantic ID，把多模态语义 ID 从静态码本升级成自适应学习，线上 GMV 也拿到了正收益。三件事其实指向同一个方向：让广告链路的每个环节，对"真实发生了什么"更敏感。

好，我们先把 TRACE 这篇展开讲一下，这是今天广告同学最该看的一篇。

做 CVR 的同学都知道一个老大难：转化是延迟的。用户点了广告，可能几分钟后下单，也可能三天后才回来。你等吧，数据就不新鲜了；你不等，就只能先把它当成没转化，也就是负样本。可后来它真转化了，你之前那条训练数据其实是错的。这个假负例问题，过去十年已经被反复打补丁，延迟分布建模、重要性采样、多窗口融合，各种姿势都试过。

TRACE 这篇的切入点很聪明。它说，你们都只盯着最终那个"转化还是没转化"的硬标签，但其实点击之后这段时间，用户一直在做事——加购、收藏、看详情、停留。这条行为轨迹本身就是很强的信号。一个点击完两分钟就加购收藏的用户，跟一个点完什么都没做的用户，他们最终转不转化的概率差得很远，你不需要非等那个最终标签。

那它怎么用这条轨迹？我用一句话讲它的核心机制：对于一个还没出现转化的样本，模型分别算两件事——如果它最后会转化，这条轨迹有多合理；如果它最后不会转化，这条轨迹又有多合理。哪边更合理，就把后验推向哪边。训练的时候也不硬标负例，而是对标签做边际化。

这里还有一个很实用的设计，叫回溯式补全器。刚点击五分钟的新样本观测太少，模型容易退化成只看静态特征。作者的办法是离线拿历史上已经跑完整条归因窗的样本，人为把轨迹截短，训练一个"只看前面一小段也能猜最终结果"的专家。上线后让这个专家给早期样本打一个软分，相当于老师来教在线模型。

放到广告系统里看，这套东西的价值在哪？在于它给你一个比硬负例更合理的监督信号。加购、收藏、停留时长这些点击后行为，很多广告系统日志里本来就有，只是没组织成轨迹去用。而且它是插件式的，能叠加在 ES-DFM、DEFER、DEFUSE 这些现有延迟反馈框架上，不用推倒重来。对 OCPX、CPA 这种对校准特别敏感的出价链路，这个收益是真金白银。

当然也有前提。你得有稳定的点击后行为日志，时间窗要切得合理；如果流量分布漂移得厉害，那个补全器还得定期重训。但整体上，这是一条很值得今年跟进的技术路线。

讲完 TRACE，我们说今天的第二篇重点，快手的 CS3。

这篇解决的是双塔召回的老问题。双塔为什么流行？因为用户塔和物品塔可以分开算，物品向量可以预先离线算好，线上用 FAISS 一查就行，又快又省。但代价也很清楚：单塔容量有限、两塔之间在最后点积之前完全不交互、跟下游精排能力差距大。过去的后期交互、蒸馏方案，要么拖延迟，要么在持续在线学习的场景里没法用。

CS3 的做法是一次性补三块短板，而且每一块都照顾了在线学习的约束。我挑最值得讲的一个机制来说——CMS，级联模型共享。

你可以这么理解：双塔天生做不了用户-物品交叉，也做不了 target attention，但下游精排能做。那能不能把精排学到的能力，蒸一点回召回？CMS 的做法是，把精排倒数第二层的输出，按用户维度、物品维度分别做 EMA 缓存，每个用户、每个物品各自攒一份"精排视角下的自我画像"。下一次召回时，用户塔和物品塔的输入里，除了原始特征，还会拼上这份从精排那边沉淀下来的向量。

这件事的意义是什么？意义是召回模型在打分的时候，已经带了一部分精排视角。论文里用 Kendall Tau 衡量召回和精排的排序一致性，线上三个场景分别提升了百分之三到百分之六。一致性上去了，你就更少出现那种召回觉得好、精排觉得一般的浪费。线上收入最高百分之八点三六的涨幅，CMS 是主力。

这里有个工程细节也值得讲一下。CTS 那些对塔向量放在参数服务器里，跟着稀疏参数一起更新；CMS 那份精排中间向量体量更大、还要跨模型共享，就单独放在一个为向量优化的 Embedding Server 里，类似 Redis 但 QPS 更高。召回请求来的时候，这些向量查询和其他特征请求并行走，p99 延迟依然在毫秒级。这种"分层缓存、各司其职"的工程思路，对任何一个大厂广告系统都有直接的参考价值。

风险也说一下。CMS 很依赖精排中间向量的稳定性，精排一旦大改版，这些缓存就会带偏召回；CTS 只用正样本更新，对热门用户热门广告可能有马太效应，冷启动场景要小心。

讲完这两篇，我们把视野稍微放宽一点，看看今天其他几篇值得留个印象的。

快手的 Adaptive Semantic ID 这篇，也在工业界很值得看。多模态 Semantic ID 最近一年特别火，思路是把每个商品用一串离散 token 表示，代替传统的 item ID。但老问题是，不同商品被分到一样的 token，也就是碰撞；还有语义漂移。这篇的贡献是把静态码本变成自适应学习，能根据下游效果动态决定哪些重叠该压、哪些该保留，线上 GMV 拿到了正收益。对做电商广告召回、尤其是商品理解的同学，这是今年绕不开的一条线。

还有一篇《Learning to Rotate》，把 Transformer 里 RoPE 的旋转流形本身当成可学习对象，让时间和语义一起参与旋转编码。听起来很数学，但实际效果是在大型社交 feed 的生成式推荐里直接改善了排序校准。对广告排序来说，校准这件事比 AUC 涨一点点更值钱，因为它直接影响出价。

再提一个——MTServe，针对生成式推荐模型在线服务的 KV 缓存复用。生成式推荐现在推理成本很高，因为每个请求都要重新编码长长的用户历史。MTServe 把大模型领域的跨请求 KV 缓存工程搬到了推荐服务链路。如果你们团队正在上生成式召回或生成式排序，这篇的工程细节可以省掉不少坑。

最后说一下今天这批论文给我的整体感受。

我开头说，今天的主判断是召回和排序之间的墙正在被重写。你回头看这几篇：TRACE 是让训练标签更贴近用户真实行为；CS3 是让召回带上精排视角；Adaptive Semantic ID 是让商品表示跟着下游效果走；MTServe 是让在线服务能承载更重的模型。这四件事合起来，其实是一句话——广告系统每一层都不再各自为政，它们在互相靠拢。

如果你今天只带走一个判断，我希望是这个：未来一两年，广告系统的竞争力不会只来自某个单点模型变强，而来自层与层之间的协同变紧。召回要懂排序的意图，排序要懂真实转化链路，在线服务要能撑住更大的模型。谁把这些"之间"的事做好，谁就能把每一次曝光的价值再往上抬一点。

今天就到这里，祝你有一个清醒、高效的上午。
