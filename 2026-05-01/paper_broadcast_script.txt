早上好，今天是五月一号。先给你一个可能反直觉的判断：非平稳并不等于要上复杂的控制算法，很多时候老老实实按历史比例分预算，反而离最优解更近。

这是今天我想先留给你的一个抓手。接下来我们会看到，今天这批论文里最值得关注的几个工作，其实都在回答同一个更大的问题：当系统越来越复杂，我们到底该什么时候加一层智能，什么时候保持克制。

先说今天的主判断。如果用一句话概括，就是——2026 年的算法工程师，正在重新学会判断"什么时候不该上大模型、不该上复杂优化"。

这个信号其实不小。你看今天三百多篇论文里，最引人注意的几篇，一个在质疑 MPC 真的比反应式 pacing 强吗，一个在质疑 LangGraph、CrewAI 这种 Agent 编排框架真的比一段精心写的 prompt 好吗，还有一个把多阶段推荐调参交给 LLM 多智能体自动跑 A/B。共同点是：都在拿控制实验去逼问"这一层复杂度到底值不值"。

换句话说，工业界的注意力，正在从"我能不能做出来"转向"我该不该上这一层"。

下面说今天值得记住的三个事件。

第一个，Learning to Spend，把季度预算分配当成一个闭环控制问题来研究 MPC。论文拿 Expedia 每年大约 68 亿美金的营销预算当背景，非常接地气。它的结论后面我会单独展开。

第二个，AgenticRecTune，来自 Google Discover 的真实线上工作。它把粗排、精排、重排的融合权重调优，交给一套五个角色的 LLM 多智能体去跑，还给这些 Agent 配了一个会自我进化的技能库。这篇后面也会重点讲。

第三个，叫 In-Context Prompting Obsoletes Agent Orchestration，标题本身就有点挑衅。它用控制实验告诉你，对于很多程序化的任务，一段设计得当的上下文 prompt，效果能打平甚至超过 LangGraph、CrewAI 这类外部编排器。对正在纠结 Agent 框架选型的团队来说，这是一个需要认真对待的反方观点。

好，进入今天和广告最相关的两篇重点。

先讲 Learning to Spend。

这篇为什么值得广告同学今天就看？因为它做了一件我们经常想做但很少认真做的事——把"到底要不要上 MPC"这个问题，用控制实验拆开回答了。

它的设定很贴近真实：一个季度 12 周，每周定一个计划预算，但平台实际花出去的钱会被拍卖和 pacing 扰动带偏；每一块钱的回报效率还可能随时间变化。你可以反应式地按历史比例摊，也可以每周解一个带前瞻的优化问题，也就是 MPC。

真正有意思的是它的分界结论。作者构造了三种场景。第一种，效率不变，MPC 和反应式几乎打平，甚至略输一点。第二种，效率是随机漂移，你预测不出规律，这时候 MPC 反而比老实的反应式还差零点三到零点四个百分点。第三种，效率有可预测的季节性衰减，比如广告疲劳、生命周期这种跨周期重复的结构，MPC 就能多赚百分之十一到百分之十四。

你听出来了吗？关键不是非平稳，关键是这种非平稳有没有结构、能不能被预测。

更值得记住的是第二个点：当结构足够强时，MPC 的瓶颈居然不是预测模型精度，而是"每周预算最多能相对上周变多少"这个平滑约束。论文里卡了一个正负 20% 的上下限，把大部分理论收益顶掉了。

这对广告团队意味着什么？意味着你如果想继续榨 pacing 的收益，优化方向可能不是继续卷 forecast 模型，而是回去和业务方谈——能不能把每周预算波动的上限从 20% 放到 30%。这是一句话的 insight，但它可能比一个季度的模型迭代更值钱。

当然，论文的实验是纯合成仿真，没有真实多渠道竞争、没有延迟转化，落地还是要小心。但它给了一个非常清爽的判据：先判断你面对的非平稳是结构性的，还是纯噪声漂移。是结构性的，上 MPC；是纯噪声，反应式加 carryover 已经逼近 oracle，别折腾。

接下来讲 AgenticRecTune。

为什么这篇对广告同构性极高？因为广告的粗排、精排、混排，本质上就是一条多阶段链路，每一层吐出一堆分数，最后靠一组融合权重和阈值合成 ECPM。这些配置几乎都是不可导的，一升级模型就要全盘重调，传统 AutoML 很难吃下来。

论文的做法，是把调参这件事拆成五个角色。Actor 负责出候选配置，Critic 负责挑刺和过滤，Online Agent 自动生成 A/B 实验代码并跑到显著性，Insight 和 Skill 这两个角色负责复盘，把每次实验的规律写回一个技能库里。下次 Actor 出方案之前，会先翻这本不断更新的"调参手册"。

我想让你记住的是其中一个机制——技能库的自进化。

它是怎么工作的？举个例子，Insight Agent 发现历史上只要多样性惩罚权重超过 0.4，整体 engagement 就会跌，这条规则就会被写进重排任务的领域知识里，同时搜索空间的上界从 0.6 直接收到 0.4。下一轮 Actor 再提案，就不会再去踩这个坑。

这相当于把算法工程师脑子里那些"经验直觉"显式地沉淀下来，变成一个会随实验积累自动增厚的 SOP。

对广告同学的启发很直接：ECPM 里那些系数，pCTR 和 pCVR 的融合方式，广告和自然结果的混排权重，频控阈值，全都可以套这套闭环。它省下的不只是调参工时，更重要的是把历史实验里那些血泪教训固化成下一次就能复用的规则。

当然风险也很明显。一是它严重依赖 Gemini 级别的强推理模型，小模型效果明显打折，成本要算清。二是广告线上 A/B 本身风险就高，Agent 对护栏的理解如果有偏差，可能直接砸收入，落地一定要有强人审和快速回滚。三是论文对 Skillhub 的具体实现披露得很少，想复现得自己补不少工程。

讲到这儿，你可能发现今天这两篇重点，其实在呼应同一个判断——都在告诉你，真正重要的不是"我用了多复杂的方法"，而是"我什么时候该用、什么时候不该用"。

再顺手提一篇外围的，给广告同学补一个点。叫 One Pass, Any Order，位置不变的 listwise 重排。它解决的卡点是：decoder-only 的 LLM 在对候选集打分时，把同一组候选换个顺序喂进去，结果就会抖。这在广告混排里是个实际问题，因为你不希望最终展示顺序是 prompt 序列化的副产品。这篇用结构化的注意力掩码，把候选之间的互相看彼此屏蔽掉，再用共享位置编码消掉位置带来的差异，让整个候选集变成真正的集合式打分。如果你们团队在探索 LLM 做重排，这是一个工程上值得抄作业的点。

最后做个总结。

今天这批论文拼起来的那个判断，我再说一次——算法工程师正在重新学会克制。MPC 只在结构性非平稳下赚钱，Agent 编排在很多场景下打不过一段好 prompt，多智能体调参真正的壁垒是那本不断长厚的技能库，而不是 Agent 本身有多聪明。

对应到广告系统，你今天可以带走三句话。第一，pacing 要不要上 MPC，先问自己面对的变化是不是可预测的结构。第二，如果结构足够强，瓶颈很可能在业务约束而不是模型。第三，真正值钱的资产，是把每次 A/B 的经验沉淀成下一次能复用的规则，这件事比多跑几个实验更重要。

好，今天的播报就到这里，祝你有一个清醒的早晨。
