## 三、今日一个 AI 知识点

### Off-Policy Evaluation（离线策略评估）：不上线就能知道新策略好不好

想象你在一个广告平台做排序。线上跑着一个老排序模型（叫日志策略），每天产生大量曝光-点击日志。现在你训了一个新模型，想知道它上线后 CTR 会不会更高——但你不敢直接上线，怕翻车。Off-Policy Evaluation 就是帮你用老日志来估计新策略效果的一整套方法。 它的核心推断流程大致是这样走的：首先你拿到一条历史日志，记录了老策略给用户展示了哪些广告、展示顺序是什么、用户点了没有。接下来你问新策略一个问题：如果是你来排，你会把这些广告排成什么样？新策略会给出每种排列的概率。然后关键的一步来了——你要算一个'重要性权重'：新策略选择这个排列的概率除以老策略选择这个排列的概率。如果新策略很喜欢这个排列而老策略不太喜欢，这条日志就会被放大；反之就被缩小。最后你把每条日志的实际收益（比如是否点击）乘以这个权重，再对所有日志求平均，就得到了新策略的估计效果。 这就是经典的 Inverse Propensity Scoring（IPS）。但它有个致命假设：老策略必须是随机的，也就是每种排列都有非零概率被选中，否则分母为零，整个估计就爆掉了。现实中很多广告系统的日志策略是确定性的——给定输入只会输出一种排列。今天候选论文里的 CIPS 方法就是为了解决这个痛点：它不是在整个排列层面做重加权，而是在'位置对'的粒度上做因果推断，通过条件独立性找到那些即使日志策略是确定性的、仍然可以估计的局部对比信号，把它们拼回全局排序的效果估计。这样一来，即使你的线上系统从来只输出一种排序，你也能在离线可靠地比较新旧策略——省掉了一次又一次昂贵且有风险的 A/B 测试。

## 六、重点论文精读

### 1. One Model, Two Markets: Bid-Aware Generative Recommendation
- **背景：** 这篇论文解决的是：生成式推荐模型原本只会按语义相关性生成内容，但真实信息流里平台还要同时考虑广告变现、广告位是否该开、以及广告主实时 bid 的变化，所以原有方法很难直接用于商业流量分配。作者提出 GEM-Rec，把自然内容和广告都放进同一条生成序列里，用控制 token 先决定当前 slot 是 \<ORG\> 还是 \<AD\>，再生成具体 item；同时在推断时把实时 bid 注入解码过程，让模型既保留生成式推荐的语义能力，又能响应市场价格变化。之所以值得看，是因为它试图把推荐生成、广告插入和竞价响应统一到一个可控框架里，还强调了更高 bid 不会让曝光概率下降这一广告机制上很重要的性质。
- **核心技术点：**
  - **点 1：统一序列建模**
    - 技术细节：论文把用户历史表示成一串带模式标记的交互，每次交互不是只记录 item，而是记录‘展示模式 + item 的语义 ID’。具体做法是在每个 item 的语义 ID 前面加一个控制 token：自然内容用 \<ORG\>，广告用 \<AD\>，然后把整段历史拼成一个自回归序列来训练 Transformer。这样模型学习的目标被拆成两步：先预测下一位更像是广告还是自然内容，再在这个模式条件下逐级生成语义 ID 的每一层 token；训练目标本质上是整条序列的似然，论文强调第一部分在学‘广告位可行性’，第二部分在学‘给定模式后的内容检索’。
    - 通俗讲解：你可以把它想成平台在每个位置先做一个小判断：这一格是正常推荐，还是适合插广告。以前很多生成式推荐模型只会回答‘推荐什么商品’，现在它先回答‘这一格该不该商业化’，再回答‘放哪个东西’。这样做的好处是，模型不会把广告 item 和自然 item 混成一个普通候选池，而是显式学到某些上下文里插广告是合理的，某些上下文里插广告会破坏体验。
    - 例子：比如用户最近连续看了几件运动鞋、跑步袜和健身手表。模型读到这段历史后，先输出当前 slot 的模式分布，可能觉得这里既有自然推荐空间，也存在商业意图，于是 \<AD\> 概率不低；如果采样到 \<AD\>，再开始生成广告 item 的语义 ID，例如先走到‘运动装备’这个大类前缀，再往下细化到‘跑鞋’子类，最后落到某个具体广告商品。反过来，如果当前上下文已经连续出现过广告，模型也可能更倾向先输出 \<ORG\>，表示这一格应该回到自然内容。
  - **点 2：两级 bid 注入**
    - 技术细节：论文的关键不是只在训练里混入广告日志，而是在推断时做两级 logit 调整。第一层是 slot 级：对 \<AD\> 这个模式 token 的原始分数做抬升，抬升量由当前有效广告候选里的最高 bid 决定，再乘一个可控系数 lambda；最高 bid 越大，模型越容易把这一位开成广告位。第二层是 item 级：一旦模式已定为 \<AD\>，模型在逐步生成语义 ID 时，会给每个候选 token 所在前缀分支加一个 bid 奖励，这个奖励取该前缀下所有有效广告里的最高 bid；也就是说，不是等完整 item 生成出来才看价格，而是在前几层前缀扩展时就把高价分支往前推。论文把这个过程称为 prefix-aware bid aggregation，并配合 beam search 做广告解码。
    - 通俗讲解：这套机制相当于把‘市场价格’插到了生成过程的两个关键岔路口。第一处岔路口是‘开不开广告位’：如果眼下广告库存里有很高价的机会，模型就更愿意开广告位。第二处岔路口是‘广告往哪条语义树枝走’：如果某个商品簇下面藏着高 bid 广告，模型会更早把搜索资源花在那条分支上，而不是平均对待所有语义上都差不多的候选。
    - 例子：假设当前时刻有三类广告候选：跑鞋类最高 bid 8，运动耳机类最高 bid 3，蛋白粉类最高 bid 6。先看 slot 决策，系统会用这批候选里的最高 bid，也就是 8，去抬高 \<AD\> 的分数；如果因此选中了 \<AD\>，接着生成语义 ID 的第一层 token 时，‘跑鞋’前缀分支拿到的分数增益最大，‘蛋白粉’其次，‘运动耳机’较低。于是 beam search 更可能先保留跑鞋相关分支，后面再在这个分支下继续细化到具体广告商品，最终输出既语义合理又更值钱的结果。
  - **点 3：单调性与保真**
    - 技术细节：论文给出一个分配规则：某个广告 item 的曝光概率，等于‘这一位被判成广告位的概率’乘上‘在广告模式下 beam search 最终选到该 item 的结果’。在这个定义下，作者证明分配单调性，也就是固定别人 bid 不变时，某个广告自己的 bid 提高，不会让它的展示概率下降。论文还给出两个结构性质：当 lambda 等于 0 时，系统退化成只依赖历史日志学到的基线生成器；同时 bid 调整只作用于广告分支，因此如果这一位走的是 \<ORG\>，自然内容之间的相对排序不被扭曲，作者把这点称为 organic integrity。
    - 通俗讲解：广告系统很怕一种事情：广告主明明出更高价，结果因为模型行为古怪，反而更不容易被展示。作者想证明自己的解码方式不会出现这种反直觉现象，所以强调‘高 bid 至少不吃亏’。另一个关键点是别为了赚钱把自然推荐也搞乱了，所以他们把价格影响限制在广告分支里，意思是：你可以改变什么时候出广告、出哪个广告，但一旦决定这格是自然内容，自然内容内部谁排前谁排后不该被 bid 污染。
    - 例子：比如广告 A 和广告 B 都在‘跑鞋’这条语义分支下，A 的 bid 从 5 提到 7。按照论文的设计，A 所在前缀分支的加分不会下降，因此 A 被 beam search 保留下来的机会只会变大或持平，最终整体曝光概率也不会更差。再看自然内容侧：如果当前 slot 最终选的是 \<ORG\>，那两双自然跑鞋商品之间的排序仍只由原始相关性决定，不会因为某家广告主突然把 bid 提高而把自然结果顺序冲乱。
  - **点 4：可控商业化旋钮**
    - 技术细节：论文用一个单一参数 lambda 控制商业化强度，它同时影响 slot 级和 item 级的 bid 增益。lambda 为 0 时，模型大致复现训练日志里的历史广告密度；lambda 增大后，广告位比例、收入都会上升，但总 NDCG 会下降，因为更多自然位被广告替换。实验里他们主要在合成市场上验证这一点：训练数据由两阶段策略生成，先按语义前缀筛广告候选，再按 bid 做概率性竞价，借此模拟‘既过用户相关性过滤，又过平台变现过滤’的广告日志。论文还展示了市场波动下的适应能力，例如对一小部分广告做 10 倍 bid shock 后，系统能在不重训的情况下迅速把曝光切向高价广告。
    - 通俗讲解：可以把 lambda 理解成平台的商业化油门。踩得轻，系统更像普通推荐，只在历史上本来就适合插广告的位置少量放广告；踩得重，系统会更积极打开广告位，并优先走向高价广告分支。这个设计很适合实际业务，因为广告市场价格天天变，平台不可能每次价格波动都重训一个大模型，所以他们希望通过推断时调旋钮就能改 ad load 和收入。
    - 例子：论文在 Steam 数据的模拟环境里报告，lambda 从 0 提到更高值后，广告占比和收入明显上升，而自然相关性指标下降较缓；在 bid shock 实验中，把 5% 商品的 bid 提高 10 倍后，lambda 为 0 时系统仍主要按历史先验行事，高价子集占展示广告的比例不高，而稍微打开 lambda 后，高价子集很快占到大多数广告曝光。需要注意，这些数值验证主要来自作者构造的合成市场，不确定在真实大型广告流量里是否能同样平滑地复现。
- **对广告的启发：** 最适合层级：检索生成层与广告位决策层；价值：最有迁移价值的是把‘是否开广告位’和‘生成哪个广告’拆成两个串联决策，再在推断时把实时 bid 分别注入这两个层面。对信息流广告、内容电商混排、商家赞助推荐都很有启发：一方面可以直接学习历史日志里的可插广告上下文，另一方面又能在不重训的前提下用实时价格调整 ad load 和广告选择。尤其适合正在尝试生成式检索或语义 ID 路线的平台，因为它利用了层级语义前缀，使 bid 能在早期搜索阶段就影响分支保留。；风险：主要风险有三点。第一，论文实验依赖合成市场，广告日志是按作者设计的两阶段策略生成的，因此对真实竞价生态、预算约束、频控、去重、转化目标、政策约束的外推存在不确定性。第二，使用前缀下最高 bid 做分支加成，可能把少数高价广告对整条语义分支的吸引力放大，实际部署时需要防止早期搜索被高价但低长期价值的分支劫持。第三，论文采用 first-price 收费并讨论了 DSIC 仍未解决，所以如果业务非常看重机制真值激励，仅靠文中方案还不够，需要再补支付规则与拍卖层设计。

### 2. TagLLM: A Fine-Grained Tag Generation Approach for Note Recommendation
- **背景：** 论文关注的是电商社区里的笔记推荐：平台上有图文或视频笔记，传统做法要么依赖固定标签池，扩展性差；要么直接让多模态大模型自由生成标签，但常常会产出又泛又散、和用户兴趣不对齐的标签。作者认为，比起只用隐式embedding，把内容变成可解释、可加权的细粒度标签，更适合线上推荐和冷启动，因此提出了一个从'按类目定义兴趣维度'到'生成标签'再到'蒸馏小模型上线'的完整方案。之所以值得看，是因为它不只讲内容理解，还讲了如何把标签真正变成推荐特征并在工业系统里拿到收益。
- **核心技术点：**
  - **点 1：兴趣手册约束**
    - 技术细节：TagLLM先不让模型盲目生成标签，而是先按类目建立一个User Interest Handbook。做法是先让LLM为每个类目扩展可能的用户关注维度，再由人工专家按平台目标做精修，形成类目专属的标签维度清单，比如鞋类会包含款式、材质、工艺、品牌、系列、风格等，游戏类会包含游戏名、类型、平台、角色、道具、玩法、版本活动等。这个手册在后续生成时充当硬指导框架，约束模型优先围绕用户真正关心的维度出标签，而不是只抓住几个高频实体词。
    - 通俗讲解：可以把这一步理解成先给模型发一本'这个类目的用户到底在看什么'的小册子。没有这本册子时，模型看到一篇鞋子笔记，可能只会吐出'Adidas''韩国''Jennie'这种宽泛词；有了手册后，它会被提醒去看鞋型、系列、配色、是否无鞋带、适合什么穿着场景这些更能区分兴趣的点。这样生成出来的标签更像'用户会拿来做决策的描述词'，而不只是内容里出现过的名词。
    - 例子：比如一篇鞋类笔记标题和正文在讲'Adidas Taekwondo女鞋，白黑配色，无鞋带，软皮，适合久穿'。如果没有手册，模型可能输出'Adidas、Jennie、韩国、运动'；用了手册后，它会沿着'基础属性、趋势风格、系列'这些维度提取，得到更聚焦的标签，如'白黑Taekwondo鞋''Adidas Taekwondo鞋''无鞋带Taekwondo鞋''Taekwondo系列'，并且后面还能给这些标签分重要度。
  - **点 2：三步标签抽取**
    - 技术细节：标签数据构造使用一个多模态CoT Extraction流程，输入包含图片或视频抽帧、标题、正文、商品描述，以及视频语音转写文本。生成流程分三步：第一步是Handbook-based Generate，模型根据手册里的维度先产出候选标签；第二步是Low-Info Tag Merge，把信息量低、重复、无关的标签合并或删除，并且如果图像和文本冲突，优先丢掉来自图像的冲突标签；第三步是Importance Rank，对保留下来的标签按1到5做重要性打分，5表示最核心。最终输出不是单纯标签列表，而是'标签:重要度分数'的结构化结果。
    - 通俗讲解：这套流程像一个先粗筛、再去噪、再排序的编辑过程。模型先把可能有用的词都找出来，然后把像'蓝色''包''好看'这种太碎、太水、太重复的表达并到更完整的标签里，最后再判断哪些标签最代表这篇笔记。这样做的关键不是让标签越多越好，而是让标签既细又准，还能区分主次，方便后面的推荐系统知道哪些点最值得用。
    - 例子：论文里给了一个包袋笔记的例子：输入有标题'夏天配蓝色迷你腋下包'、正文描述、商品信息'ROSEDAWN保龄球包 PU肩挎斜挎手提包'。模型先根据穿搭类手册产生候选词，再把低信息表达收敛后，输出类似'Blue shoulder bag':5、'ROSEDAWN shoulder bag':5、'Bowling bag':4、'Summer outing bag':3、'High-value crossbody bag':3。你可以把它想成：系统先看懂这是个什么包，再决定哪些特征最代表这条内容，而不是把所有提到的词都平铺进去。
  - **点 3：蒸馏小模型上线**
    - 技术细节：为了降低线上推理成本，作者没有直接拿大模型做在线生成，而是设计了Tag Knowledge Distillation。训练分两阶段：先用SFT数据集做监督微调，让小模型学会按照目标格式生成标签；再用DPO进一步优化，具体做法是先用SFT后的模型在DPO池上生成答案，再把生成结果和参考答案做成偏好对，只保留那些被判定为劣于参考答案的样本，参考答案作正样本、模型生成作负样本。评估时使用MLLM-as-a-judge，包含两类分数：Basic Attribute Score看真确性、覆盖度、重要性，每项0或1；Pairwise Score做A和B的两两比较，打成从正2到负2后求平均，并通过交换位置做双向评估来减轻位置偏置。
    - 通俗讲解：直觉上，这是把'会做题的大老师'的标签生成习惯，尽量教给便宜、快的小模型。第一阶段先让小模型学会基本格式和基本思路，第二阶段再专门纠正它'虽然会写，但写得不够像参考答案'的地方。因为标签生成是开放题，不太适合只靠字符串精确匹配，所以作者让一个更强的模型像评审老师一样，从'有没有胡说、有没有覆盖手册维度、有没有抓住重点'这些角度打分。
    - 例子：假设小模型看到一篇篮球鞋笔记，生成了'Curry、绿色、勇士'，而参考答案是'Curry':5、'Golden State Warriors':4、'绿色':3。评审模型会先看小模型输出是否真实、是否覆盖了应该关注的维度、是否抓住重点；如果整体不如参考答案，这条样本就进入DPO训练，小模型被进一步推向更像参考答案的输出风格。论文结果显示，两阶段训练后4B级小模型的表现已经接近大模型评审参考，说明这条蒸馏路线对线上部署是实用的。
  - **点 4：标签进推荐链路**
    - 技术细节：生成好的细粒度标签最终被转成两类推荐特征。第一类是User Profile：对用户历史交互过的笔记，收集其标签集合，并按交互行为给标签加分，例如点击、点赞、收藏分别增加不同权重，形成带分值的用户兴趣标签画像。第二类是Tag Embedding：把单篇笔记的全部标签拼接后，用预训练编码器映射成向量，作为内容特征。在线架构里，这些标签特征与原有用户静态特征、物品静态特征一起送入User Tower和Item Tower训练，用BCE和对比学习联合优化；推理时由User Tower生成用户向量，在Item ID索引里做Top-K召回。
    - 通俗讲解：这里的核心不是'生成标签就结束了'，而是把标签真正喂给推荐系统。用户看过什么内容、点过什么内容，会沉淀成一串带权重的兴趣标签；而每篇新笔记也会有一串结构化标签和一个标签向量。这样系统做匹配时，不只是靠黑盒内容向量说'这两个像'，还可以更明确地知道这个用户最近偏爱'白黑Taekwondo鞋''无鞋带''复古运动风'，于是更容易把相似的新内容召回出来。
    - 例子：比如某用户最近点击了多篇带'Adidas Taekwondo鞋''白黑配色''无鞋带''Jennie风格'标签的笔记，还对其中一些点了赞和收藏，那么这些标签在用户画像里的分值会逐步升高。此时一篇新发笔记虽然还没有多少行为数据，但TagLLM已经先生成了类似标签和对应向量，召回阶段就能把它和这个用户连上，解释上也更直观：不是因为一个模糊向量接近，而是因为标签层面对上了。
  - **点 5：效果与边界**
    - 技术细节：线上A/B测试显示，TagLLM带来了平均每用户观看时长加0.31%、平均每用户互动加0.96%、GMV加0.15%、UVCTR加0.06%、每用户有效点击加0.21%。冷启动场景下，针对24小时内新发布笔记，相比已有标签倒排方法，PVCTR提升32.37%，PVIR提升46.91%。不过论文也明确指出两类失败案例：一类是Reasoning Hallucination，即模型在信息不完整时自己补错事实；另一类是User Misdirection，即原始内容本身写错或图文不一致，导致模型被误导。
    - 通俗讲解：这说明细粒度标签最强的地方，不一定是把成熟热门内容再提一点点，而是帮助新内容更早被看见。因为新笔记没有足够点击和互动，系统很难只靠行为信号理解它，但标签能提前告诉系统'它到底讲了什么'。不过这条路也有代价：如果模型理解错了，或者用户原文就写错了，那么错误标签会被放大到推荐链路里，所以线上一定要有质量控制。
    - 例子：论文举的错误例子里，有一篇车内容提到Integra和G70，但模型错误地补成了'Toyota Integra''Mazda G70'，这就是推理型幻觉；另一篇杯子内容里用户自己打了'#DisneyStitch'，但图片其实是'Toy Story三眼仔'，模型就可能被错误标签带偏。如果这些错误标签直接进召回，可能把内容送给完全不相关的人群，所以这类系统上线时最好配套事实校验、知识检索或低置信度拦截。
- **对广告的启发：** 最适合层级：内容理解与冷启动召回层；价值：对广告最有迁移价值的是把素材、商品详情页、达人内容、落地页统一转成'细粒度可解释标签+重要度'，再把这些标签做成用户画像和物料特征。这样可用于冷启动广告召回、创意理解、相似创意扩展、兴趣定向解释和检索增强，尤其适合电商广告、内容广告和原生推荐广告。一个直接迁移方式是先按广告类目建立'投放兴趣手册'，再生成如风格、卖点、价格带、人群、使用场景、材质、品牌系列等标签，用于补足新广告几乎没有行为数据时的匹配能力。；风险：风险主要有三类。第一，广告比社区笔记更容易出现夸张文案、隐含卖点和合规敏感表达，开放式生成可能产生幻觉或错误归因；第二，论文中的推荐收益来自Dewu社区笔记场景，迁移到广告点击和转化链路时，收益大小存在不确定性，不能直接照搬；第三，论文没有在摘录中详细公开用户画像加分的具体权重、标签编码细节和线上特征工程全部配置，因此复现广告效果时需要重新校准标签质量阈值、行为权重和召回融合策略。
